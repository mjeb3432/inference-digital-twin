## Problem Statement
Build an open source research platform that maps inference engineering decisions from hardware to software to operations, then predicts user-visible outcomes. The product should let someone change chip choice, interconnect, runtime strategy, and orchestration assumptions, then immediately inspect latency, concurrency, throughput, cost, and energy tradeoffs.

## What Makes This Cool
Most tools are deep in one layer and blind to the others. This project is exciting because it lets operators and researchers test cross-layer tradeoffs in one place. A user should be able to answer, "If I move from A100 to H200, change parallelism strategy, and alter runtime settings, what actually happens to TTFT and system-level capacity?"

## Constraints
- Primary goal is research quality, not a lightweight demo.
- Must be open source and transparent enough for others to challenge assumptions.
- Primary audiences are data center operators and ML/AI researchers, with students as a tertiary audience.
- Scope spans hardware, interconnect, runtime, orchestration, and operational outcomes.
- First release still needs a bounded scope so it can ship.

## Premises
1. The right wedge is cross-layer decision support, not another single-layer benchmark dashboard.
2. The long-term architecture should be a modular digital twin engine with pluggable hardware, runtime, and orchestration models.
3. Even in a research-first architecture, early versions must start with a constrained scenario matrix for calibration and trust.
4. Core outputs must include TTFT, TPOT/TPS, concurrency, cost, and power envelope per scenario.
5. Differentiation comes from exposing interaction effects across layers, not just listing isolated benchmark scores.
6. Distribution must be open and reproducible: public web app, open assumptions, versioned scenario definitions, and documented methodology.

## Approaches Considered
### Approach A: Research MVP Explorer
Build a constrained web app with explicit estimation formulas and a small scenario matrix.

- Effort: M
- Risk: Low
- Pros: ships quickly, easy to get feedback, forces clarity on assumptions.
- Cons: lower realism initially, limited breadth, may feel lightweight to advanced operators.

### Approach B: Inference Digital Twin Platform
Build a modular engine with pluggable simulation/estimation modules for silicon, interconnect, runtime, scheduler, and energy layers, then expose through an interactive UI.

- Effort: XL
- Risk: High
- Pros: strongest long-term architecture, extensible research platform, best fit for deep operator and researcher use cases.
- Cons: high upfront complexity, slower first public artifact, greater calibration burden.

### Approach C: Scenario DSL + Explorer
Define scenarios as structured config files and drive both interactive visualizations and narrative reports from the same schema.

- Effort: L
- Risk: Medium
- Pros: reproducible research workflow, easy sharing via git, good balance of rigor and speed.
- Cons: schema design complexity, extra onboarding cost for non-technical users.

## Recommended Approach
Proceed with **Approach B** (selected) and execute it in phases so ambition does not block shipping:
- Phase 1: Narrow digital twin core for one model family and a small hardware/runtime matrix.
- Phase 2: Expand modules for interconnect and orchestration fidelity.
- Phase 3: Add richer calibration and validation datasets.
- Phase 4: Broaden vendor/model coverage and publish comparative studies.

This preserves the research-grade architecture while reducing delivery risk.

## Success Criteria
- Users can model at least 20 representative scenarios spanning hardware, runtime, and orchestration dimensions.
- For baseline scenarios, predicted trend directions match known empirical behavior (for example, memory bandwidth increases do not worsen throughput).
- The app returns TTFT, TPOT/TPS, concurrency, cost, and power outputs for every valid scenario.
- At least 3 external researchers or operators can reproduce published scenarios from repository artifacts and get matching results.
- Public documentation explains assumptions, equations, calibration limits, and known failure modes.

## Engineering Locks (/plan-eng-review)

### Scope Lock
- Phased Approach B accepted: v1 core first, then interconnect/orchestration depth in later phases.

### Architecture Decisions
1. Versioned contracts are mandatory: `ScenarioSpec v1`, `ModuleInput/Output v1`, `PredictionReport v1`.
2. Deterministic calibration pipeline is mandatory with versioned coefficient artifacts.
3. Mandatory provenance in every report: scenario hash, artifact ids, module versions, timestamp, and commit id.
4. Async job architecture is mandatory for compute isolation: API enqueue -> worker run -> stored report by run id.
5. Distribution architecture locked: hosted app + versioned Docker image + downloadable scenario packs + release metadata.

### Code Quality Decisions
1. Hard dependency direction: scenario -> orchestrator -> modules -> report; modules cannot import each other.
2. Canonical error taxonomy required: `validation`, `data_missing`, `calibration_out_of_range`, `compute_timeout`, `internal`.
3. Assumptions registry required: versioned, machine-readable, referenced by module/report metadata.
4. Inline ASCII diagrams required in complex implementation files.

### Test Strategy Locks
1. Full test pyramid required in v1: contract + module + integration + E2E.
2. Per-metric tolerance policy required with deterministic seed strategy.
3. Golden scenario regression suite required as PR gate.

### Performance Decisions
1. Content-addressed cache required: key = scenario hash + artifact version + module versions.
2. Queue backpressure policy required: max depth, per-user concurrency caps, TTL, cancellation, degraded-mode UX.
3. Stage-level telemetry required: queue wait, per-module latency, cache hit rate, error class rate, p50/p95 e2e latency, cost/run.

## Failure Modes and Handling
- Queue outage: covered by tests, explicit retryable error, user-visible message.
- Calibration artifact incompatibility: covered by tests, hard reject with compatible-version guidance.
- Module timeout: covered by tests, canonical `compute_timeout` with run-state transition and user messaging.
- Scenario hash instability: covered by tests, deterministic canonicalization required.
- Regression drift beyond tolerance: covered by golden-suite PR gate, block merge on unexplained drift.

## NOT in scope (explicitly deferred)
- Full-fidelity interconnect simulation in v1: deferred to keep first release shippable.
- Vendor-complete hardware catalog in v1: start with constrained matrix for calibration quality.
- Automatic model-routing/go-to-market optimization engine: valuable but not required for first research release.
- Multi-tenant auth/billing: defer until there is sustained external usage demand.

## What already exists
- Public benchmark baselines (MLPerf, runtime docs) exist and should be reused for calibration inputs.
- Existing single-layer tools (runtime benchmarks, DCIM telemetry) partially solve sub-problems; this plan composes them instead of rebuilding their domains.
- No local repository code was detected in this workspace, so implementation reuse decisions are architecture-level for now.

## Worktree Parallelization Strategy

| Step | Modules touched | Depends on |
|------|------------------|------------|
| Contract + schema foundation | contracts/, schema/, validation/ | — |
| Calibration pipeline | calibration/, artifacts/, ingestion/ | contracts |
| Orchestrator + queue + worker | api/, orchestrator/, queue/, workers/ | contracts |
| UI explorer + run/report pages | ui/, api-client/ | orchestrator |
| Caching + telemetry | cache/, observability/, workers/ | orchestrator |
| CI golden regression gates | ci/, tests/, fixtures/ | contracts, calibration |

Lane A: contract/schema foundation -> calibration pipeline (sequential, shared contracts/)
Lane B: contract/schema foundation -> orchestrator/queue/worker -> caching/telemetry (sequential, shared orchestrator/)
Lane C: contract/schema foundation -> CI golden regression gates (mostly independent after contracts)
Lane D: UI explorer/run/report flows (starts after orchestrator API contracts stabilize)

Execution order: Launch Lane A + Lane B + Lane C in parallel after contract foundation lands. Launch Lane D once orchestrator endpoints are stable.

Conflict flags:
- Lanes A/B/C all touch contracts/ early; treat contract work as a short sequential pre-step.
- Lanes B and D share API contract surface; freeze API schema before parallel UI work.
