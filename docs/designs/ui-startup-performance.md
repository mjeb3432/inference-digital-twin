## Problem Statement

The Forge's startup UX had two friction points:
1. A 10-second cinematic intro before the app was usable
2. FastAPI service initialization blocked the desktop app from responding until complete

This branch addresses both. The intro is cut to 3.2s. Services warm in a background thread so the UI starts rendering immediately.

The remaining gap: the "boot shell" placeholder still shows a loading screen feel, which conflicts with DESIGN.md's "calm control-room energy — instruments don't show spinners."

## Vision

> **Note: Vision is aspirational north star. Implement only the items listed in Accepted Scope below.**

### 10x Check

The Forge should feel like opening a Bloomberg Terminal — the moment you launch it, your last scenario is right there, rendered, ready to simulate. No loading screens, no spinners. The app remembers your work across sessions (auto-save is a future phase, not this branch). When services are initializing, a single ambient indicator tells you without interrupting the experience. The cinematic intro is available on demand via demo mode; in daily use the app goes straight to work.

## Accepted Scope (added to this plan)

### 1. localStorage last-session restore
Renderer context: PyQt6 `QWebEngineView` with default profile — localStorage is supported and persisted to disk by Chromium under the default storage path. No fallback needed for this context.

On launch, `forge.js` reads `localStorage.getItem("idt_session_v1")` (key: `idt_session_v1`). Stored value is a JSON object with a top-level `schemaVersion` field that mirrors `facilityState.scenario.schemaVersion` (already present in the state object at `facilityState.scenario.schemaVersion`, e.g., `"1.0.0"`).

**Startup flow:**
1. If key missing → first launch → show boot shell → render canvas after hydration.
2. If key present and `parsed.schemaVersion === facilityState.scenario.schemaVersion` → set `facilityState` from stored value → skip boot shell → call `renderAll()` directly.
3. If key present but parse fails or version mismatch → stash old blob via `localStorage.setItem("idt_session_v1_backup_" + Date.now(), rawBlob)` before clearing (timestamp key preserves each backup independently — second upgrade does not overwrite first backup) (preserves data for future migration), then `localStorage.removeItem("idt_session_v1")`, `console.warn("idt: session state cleared — schema version mismatch")` → call `resetFacilityState()` → show boot shell with schema-mismatch notice → proceed with defaults. **Schema-mismatch notice:** When entering the boot shell via this path (not clean first launch), render a secondary `.canvas-boot-copy` pill below the "INITIALIZING" line: `(i) Previous session cleared — schema updated. Your config has been backed up.` This makes the stash behavior visible to the user. Set a flag (e.g., `const showMismatchNotice = true`) before calling `renderBootShell()` and read it inside to conditionally render the notice.

**Write:** On every state mutation that changes `facilityState`, write `idt_session_v1` via a trailing 1-second debounce (trailing = write fires 1s after the last change, not the first). Wire to `recalcAll()` — the existing top-level state update function.

Wrap the `localStorage.setItem` call in try/catch. On `QuotaExceededError`, log `console.error("idt: localStorage quota exceeded — session not saved")`. Silent failure is worse than a logged error.

**closeEvent flush:** In the PyQt6 desktop window (e.g., `MainAppWindow`), override `closeEvent()` to flush any pending debounce write immediately before the window closes. Use the callback form of `runJavaScript` with a 500ms `QTimer` guard — if the JS call does not complete within 500ms, log a warning and allow the close to proceed. This prevents the app from hanging on exit if the renderer is wedged:
```python
def closeEvent(self, event):
    self._flush_done = False
    def _on_flush(result):
        self._flush_done = True
        event.accept()
    self._browser.page().runJavaScript(
        "try { localStorage.setItem('idt_session_v1', JSON.stringify(facilityState)); } catch(e) {}",
        _on_flush
    )
    QTimer.singleShot(500, lambda: event.accept() if not self._flush_done else None)
```
This closes the 1-second data-loss window on app quit without risking a hung exit.

**Boot shell removal:** When `scheduleInitialHydration()` completes (canvas rendered), `el.constructionCanvas.innerHTML` is overwritten by the real canvas render — the boot shell div is replaced, not toggled. On localStorage restore path, `renderBootShell()` is never called.

**Boot shell timeout fallback:** If `scheduleInitialHydration()` never completes (JS error, race condition), the boot shell would stay on screen indefinitely. Add a 10-second timeout that replaces the boot shell with error copy:
```javascript
setTimeout(() => {
  if (el.constructionCanvas.querySelector('.canvas-boot-shell')) {
    el.constructionCanvas.innerHTML = '<div class="canvas-boot-error"><span>&#x2717;</span><span>INITIALIZATION FAILED — Restart the app. Logs: logs/app.log</span></div>';
  }
}, 4_000);
```
Fire once; does not repeat. **Timeout reduced from 10s to 4s** (outside voice finding #8): fail fast — 10s of blank waiting is user-hostile, the same reason the intro was cut from 10s to 3.2s.

**`.boot-error` / `.canvas-boot-error` CSS spec (design review addition):** Styled to match the existing `.canvas-boot-copy` dark glass aesthetic. IBM Plex Mono 13px. Color `#B91C1C`. Border `1px solid rgba(185,28,28,0.4)`. Background `rgba(7, 18, 31, 0.82)`. Same padding (14px 18px) and box-shadow as `.canvas-boot-copy`. Centered in canvas via `display: grid; place-items: center` on the parent container. No spinner — static, terminal state.

### 3. Service readiness indicator (ambient dot)
Location: persistent overlay div as sibling to `el.constructionCanvas` (not a child — see Reviewer Concern #3). Implemented by extending the existing `.led` component with state modifiers (no new component). States: `.led.led--warming` = pulsing amber = warming/error-during-fetch, `.led.led--ok` = solid green = service ready, `.led.led--error` = solid red = service error (health returns `"error"`). Unknown health status values (future API evolution) are treated as warming — dot stays amber, polling continues.

**Insertion timing:** Create the overlay div at the top of `init()`, unconditionally, before the localStorage restore check and before calling either `renderBootShell()` or `scheduleInitialHydration()`. This guarantees the dot is mounted regardless of which startup path runs.

**Boot shell suppression:** On the first-launch path (no localStorage state), the overlay starts with `opacity: 0; visibility: hidden` while the boot shell is visible — suppresses dual loading signals. The overlay becomes visible (`opacity: 1; visibility: visible`) when `renderCenterCanvas()` fires (same moment the live canvas replaces the boot shell). On the localStorage restore path, the overlay is visible immediately (no boot shell is shown).

**Label:** The overlay contains a text label `SERVICES` in IBM Plex Mono 12px `#94A3B8` (muted text from DESIGN.md), positioned to the left of the dot. On `status === "error"`, the label changes to `SERVICE ERROR` in `#B91C1C`. Label shares the same opacity transition as the dot when fading out.

**Dot visual spec:**
- Outer wrapper: 24×24px div, `pointer-events: none` — provides adequate hover/touch target without changing dot size
- Inner dot: 8px circle via the `.led` component (extends existing design language)
- Colors (DESIGN.md semantic tokens): warming `#B45309` (warning), ok `#15803D` (success), error `#B91C1C` (error)
- `.led.led--warming`: shares the existing `.led.live` pulse animation (same keyframe, same visual metaphor)
- `.led.led--ok`: no animation; `background-color: #15803D`; add `transition: background-color 200ms ease-out` for perceptible amber→green swap before fade
- `.led.led--error`: no animation; `background-color: #B91C1C`; static

**Position:** `position: absolute; top: 12px; right: 12px; z-index: 10` within the overlay div. The overlay must have `position: relative` or `position: absolute` covering the same area as `el.constructionCanvas`. The overlay itself has `pointer-events: none` so it never blocks canvas interactions.

**Accessibility:** Overlay div: `role="status"`, `aria-live="polite"`, `aria-label="Service status: warming"`. Inner `.led` span: `aria-hidden="true"` (visual only). On each state change, update `aria-label`: warming → `"Service status: warming"`, ok → `"Service ready"`, error → `"Service error — restart the app"`.

**Canvas background (rendering gap fix):** Add `background: var(--surface-base, #0F172A); min-height: 100%` to `#constructionCanvas` in `forge.css`. Ensures the canvas container renders as an intentional dark surface during the gap between localStorage restore and `renderAll()` completing — no flash of unstyled content.

**Polling:** `const pollId = setInterval(checkHealth, 500)`. On fetch network error (server not yet bound), catch and keep pulsing amber — expected behavior during first 1-2s of startup. On `status === "ok"`: `clearInterval(pollId)`, swap to `.led.led--ok`, then fade after 2000ms. On `status === "error"`: swap to `.led.led--error`, change label to `SERVICE ERROR`, `clearInterval(pollId)` — no further polling. Unknown status: keep amber, continue polling. Fire-and-forget once resolved; polling does not resume after ok.

**Fade timing:** `setTimeout(() => { dot.style.opacity = '0'; }, 2000)` fires when `status === "ok"` is first received. The CSS fade is `transition: opacity 300ms ease-in` (exit animation = `ease-in` per DESIGN.md). After the transition completes, set `visibility: hidden` via a `transitionend` listener. [Amended from plan: `ease-out` → `ease-in` per DESIGN.md exit convention; `visibility: hidden` added to remove element from tab order after fade.]

**Pulse animation spec:** `.led.led--warming` uses the existing `.led.live` pulse animation. If a separate `@keyframes` is needed: `@keyframes service-pulse { 0%, 100% { opacity: 1 } 50% { opacity: 0.35 } }` with `animation: service-pulse 1.6s ease-in-out infinite`. 1.6s is an ambient "breathing" rhythm distinct from DESIGN.md's transition scale (which governs single-state changes, not infinite loops).

### 5. Bundled demo scenario (first-launch path)

On first launch (no localStorage), instead of showing the boot shell with an empty canvas, load a built-in H100 demo scenario. The developer sees a pre-filled simulation configuration and — once the service warms — live results for a realistic H100 cluster. No configuration required. This is the Champion-tier TTHW unlock: from "clone + pip install" to "real inference numbers" without touching a picker.

**Startup timing instrumentation:** Add `console.time('idt:startup')` at the very top of `init()` and `console.timeEnd('idt:startup')` at the end of `renderAll()` (after the final canvas paint). This gives any contributor a measurable baseline for startup performance in DevTools. Label clearly with `idt:` prefix to avoid collisions with framework timing.

**Startup flow change:**

| Path | Before (Section 1 base) | After |
|------|--------------------------|-------|
| No localStorage | `renderBootShell()` → wait for hydration | `facilityState = structuredClone(DEMO_SCENARIO_H100)` → `renderAll()` directly |
| localStorage present | restore from storage → `renderAll()` | unchanged |

The demo scenario uses the same code path as localStorage restore. No new branching — the only difference is the source of `facilityState`.

**`DEMO_SCENARIO_H100` constant:** Defined at the top of `forge.js`. Represents an 8-GPU H100 node cluster with InfiniBand fabric, tensor parallelism × 8, San Jose site, LLM inference workload, BF16 precision. Exact field names and structure must match the live `facilityState` schema (read from the existing `facilityState` initialization in `forge.js`). The `schemaVersion` field must equal `facilityState.scenario.schemaVersion` so the localStorage restore logic does not clear it on the next launch.

**First-run flag — `_isDemoData`:** `DEMO_SCENARIO_H100` includes `_isDemoData: true`. The debounce write checks this flag before writing to localStorage: if `facilityState._isDemoData === true`, skip the write. `recalcAll()` sets `facilityState._isDemoData = false` on its first call triggered by a user mutation (any picker change). This prevents the unmodified demo from being written to localStorage and incorrectly restored as "the user's session" on second launch.

Example schematic (implementer: derive actual field names from `facilityState` init):
```javascript
const DEMO_SCENARIO_H100 = {
  schemaVersion: "1.0.0",      // must match facilityState.scenario.schemaVersion
  site: { city: "San Jose", workload: "llm_inference" },
  power: { /* sensible defaults */ },
  cooling: { /* sensible defaults */ },
  compute: { gpuSku: "H100_SXM5", gpuCount: 8, nodeCount: 8 },
  networking: { topology: "leaf_spine", fabric: "infiniband", intraNode: "nvlink" },
```

**Session persistence:** The mutation debounce fires 1s after the developer first interacts with any picker. From that point, the user's state (not the demo data) is what gets persisted. On second launch, localStorage restore takes over — the developer's own configuration is restored, not the demo.

**Boot shell interaction:** The boot shell timeout (Section 1) only fires if `scheduleInitialHydration()` is still pending. On the demo path, `renderAll()` is called directly — the boot shell is never rendered and the timeout never needs to fire. Remove the boot shell timeout guard from the demo path: `if (el.constructionCanvas.querySelector('.canvas-boot-shell'))` already handles this safely since the element won't exist.

### 6. File logging

Add Python logging to capture FastAPI and service initialization errors to a file. This enables the boot-error copy in Section 1 to reference a real, findable log path — converting a dead-end error message into a diagnostic entry point.

**Setup location:** `app/services.py`, at the start of `AppServices._initialize()`, before uvicorn startup.

**Log file path:** `{project_root}/logs/app.log`, where `project_root = Path(__file__).parent.parent`. Create the directory with `mkdir(parents=True, exist_ok=True)`. Add `logs/` to `.gitignore`.

**Python setup:**
```python
import logging
from pathlib import Path

def _setup_file_logging(project_root: Path) -> Path:
    log_dir = project_root / "logs"
    log_dir.mkdir(parents=True, exist_ok=True)
    log_file = log_dir / "app.log"
    handler = logging.FileHandler(log_file, encoding="utf-8")
    handler.setFormatter(logging.Formatter("%(asctime)s %(levelname)s %(name)s: %(message)s"))
    logging.root.addHandler(handler)
    logging.root.setLevel(logging.INFO)
    return log_file
```

Call at the top of `_initialize()`, and add startup timing:
```python
import time
_log_file = _setup_file_logging(Path(__file__).parent.parent)
_t0 = time.monotonic()
logging.info("AppServices._initialize() starting, log: %s", _log_file)
# ... (existing initialization code) ...
logging.info("AppServices: service ready in %.2fs", time.monotonic() - _t0)
```
The `"service ready in Ns"` log line surfaces startup performance regression in the log file that evaluators and contributors will actually read. This is the Python-side complement to the `console.time('idt:startup')` JS instrumentation.

**launch.bat stderr capture:** Append stderr from the Python process to the same log file so crashes that happen before Python logging is set up are also captured:
```batch
.venv\Scripts\python.exe -m desktop.desktop_main 2>> logs\app.log
```
Add this line to both the first-run and repeat-run paths in `launch.bat`.

**Boot-error copy update (Section 1):** Replace:
```
✗ INITIALIZATION FAILED — Restart the app to retry.
```
with:
```
✗ INITIALIZATION FAILED — Restart the app. Logs: logs/app.log
```
The `logs/` path is relative to the project folder — the same folder the developer cloned into.

## Reviewer Concerns (for implementing engineer)

4. **facilityState reset on mismatch:** The code should call `resetFacilityState()` (or equivalent) when clearing the localStorage key. This is implied but not stated.

## Implementation Notes

- `AppServices._initialize()` uses `threading.Lock` with a double-check before and inside the lock. See `app/services.py` — the implementation is correct. Do not refactor without understanding the double-check pattern (first check avoids lock contention on every call post-init; second check inside lock handles the race where two threads both pass the first check).
- The boot shell placeholder (`canvas-boot-shell`) is rendered by `renderBootShell()` only on first launch (no localStorage state). On subsequent launches, `scheduleInitialHydration()` runs with pre-hydrated `facilityState` and overwrites `el.constructionCanvas` directly with the live canvas.
- `runtime_or_503()` in `app/api/routes.py` applies to all data routes (`/api/runs`, `/api/validate-scenario`, `/api/runs/{id}`, `/api/reports/*`). The `/api/health` and `/forge` routes intentionally do NOT go through `runtime_or_503()` — they must always respond regardless of service state.
