## Problem Statement

The Forge's startup UX had two friction points:
1. A 10-second cinematic intro before the app was usable
2. FastAPI service initialization blocked the desktop app from responding until complete

This branch addresses both. The intro is cut to 3.2s. Services warm in a background thread so the UI starts rendering immediately.

The remaining gap: the "boot shell" placeholder still shows a loading screen feel, which conflicts with DESIGN.md's "calm control-room energy — instruments don't show spinners."

## Vision

> **Note: Vision is aspirational north star. Implement only the items listed in Accepted Scope below.**

### 10x Check

The Forge should feel like opening a Bloomberg Terminal — the moment you launch it, your last scenario is right there, rendered, ready to simulate. No loading screens, no spinners. The app remembers your work across sessions (auto-save is a future phase, not this branch). When services are initializing, a single ambient indicator tells you without interrupting the experience. The cinematic intro is available on demand via demo mode; in daily use the app goes straight to work.

## Accepted Scope (added to this plan)

### 1. localStorage last-session restore
Renderer context: PyQt6 `QWebEngineView` with default profile — localStorage is supported and persisted to disk by Chromium under the default storage path. No fallback needed for this context.

On launch, `forge.js` reads `localStorage.getItem("idt_session_v1")` (key: `idt_session_v1`). Stored value is a JSON object with a top-level `schemaVersion` field that mirrors `facilityState.scenario.schemaVersion` (already present in the state object at `facilityState.scenario.schemaVersion`, e.g., `"1.0.0"`).

**Startup flow:**
1. If key missing → first launch → show boot shell → render canvas after hydration.
2. If key present and `parsed.schemaVersion === facilityState.scenario.schemaVersion` → set `facilityState` from stored value → skip boot shell → call `renderAll()` directly.
3. If key present but parse fails or version mismatch → `localStorage.removeItem("idt_session_v1")`, `console.warn("idt: session state cleared — schema version mismatch")` → show boot shell → proceed with defaults.

**Write:** On every state mutation that changes `facilityState`, write `idt_session_v1` via a trailing 1-second debounce (trailing = write fires 1s after the last change, not the first). Wire to `recalcAll()` — the existing top-level state update function.

**Boot shell removal:** When `scheduleInitialHydration()` completes (canvas rendered), `el.constructionCanvas.innerHTML` is overwritten by the real canvas render — the boot shell div is replaced, not toggled. On localStorage restore path, `renderBootShell()` is never called.

**Boot shell timeout fallback:** If `scheduleInitialHydration()` never completes (JS error, race condition), the boot shell would stay on screen indefinitely. Add a 10-second timeout that replaces the boot shell with error copy:
```javascript
setTimeout(() => {
  if (el.constructionCanvas.querySelector('.canvas-boot-shell')) {
    el.constructionCanvas.innerHTML = '<p class="boot-error">INITIALIZATION FAILED — restart the app.</p>';
  }
}, 10_000);
```
Reuses the same error copy as the ambient dot error state. Fire once; does not repeat.

### 2. Skip key for desktop intro
In both `SpaceTitleScreen` and `WbrTitleScreen`, override `keyPressEvent(self, event: QKeyEvent)` and `mousePressEvent(self, event)` on the `QGraphicsView` subclass itself (already the top-level widget — no overlay needed). Both call `self._finish()` immediately (same function used when the timer expires). No fade — immediate jump to The Forge. All keys trigger skip; no filtering by key code.

### 3. Service readiness indicator (ambient dot)
Location: persistent overlay div as sibling to `el.constructionCanvas` (not a child — see Reviewer Concern #3). 8px circle via CSS. States: pulsing amber = warming/error-during-fetch, solid green = ok, solid red = service error (health returns `"error"`).

Position: `position: absolute; top: 12px; right: 12px` within the overlay div. The overlay must have `position: relative` or `position: absolute` covering the same area as `el.constructionCanvas`.

Polling: `const pollId = setInterval(checkHealth, 500)`. [Amended from 2000ms: 500ms means the dot reflects real service state within 0.5s at negligible overhead on a local endpoint.] On fetch network error (server not yet bound), catch and keep pulsing amber — this is expected behavior during first 1-2s of startup. On `status === "ok"`: call `clearInterval(pollId)`, transition dot to green, then fade it out after 2000ms via a 300ms CSS `opacity` transition (`transition: opacity 300ms ease-out`). On `status === "error"`: set dot to red, call `clearInterval(pollId)` — no further polling (service failed). Fire-and-forget once resolved; polling does not resume after ok.

**Fade timing:** `setTimeout(() => { dot.style.opacity = '0'; }, 2000)` fires when `status === "ok"` is first received. The 300ms CSS transition handles the visual fade.

### 4. Demo mode launcher
`launch-demo.bat` in repo root calls `launch.bat` with `IDT_FAST_INTRO=0` set in the environment (`set IDT_FAST_INTRO=0` then `call launch.bat`). For packaged `.exe` builds, the same env var is passed — no code change needed to the application; `os.getenv("IDT_FAST_INTRO", "1")` is already read in `desktop/screens/space_title_screen.py`. No code changes required; new batch file only. `wbr_title_screen.py` does not read `IDT_FAST_INTRO` — its duration is independent and controlled by its own timer constant.

## Reviewer Concerns (for implementing engineer)

1. **wbr_title_screen.py skip-key:** Section 2 mentions both screens but the demo-mode note says WBR has its own independent timer. Confirm: does WBR also get keypress/click skip? Probably yes — add one line to Section 2.
2. **idt_session_v1 JSON shape:** Entire `facilityState` object is stored (implied but not stated). The value is the full `facilityState` JSON.
3. **Dot survival after innerHTML replacement:** The ambient dot must live in a persistent overlay div sibling to `el.constructionCanvas`, not inside it. A child of `el.constructionCanvas` gets wiped when `innerHTML` is replaced during canvas render.
4. **facilityState reset on mismatch:** The code should call `resetFacilityState()` (or equivalent) when clearing the localStorage key. This is implied but not stated.
5. **Debounce trigger point:** Wire the debounce write to `recalcAll()` — the existing top-level state update function in `forge.js`. This is the minimal-diff approach; no new setter abstraction needed.
6. **Fade timing:** `setTimeout(() => { dot.style.opacity = '0'; }, 2000)` fires when `status === "ok"` is first received. The 300ms CSS transition handles the visual fade.
7. **launch-demo.bat env propagation:** Windows `set` scoping means child processes inherit the var if launched in the same shell context. Verify `launch.bat` does not spawn a new `cmd.exe` subprocess that would drop the env.
8. ~~**localStorage profile (CRITICAL)**~~ **RESOLVED:** `desktop/screens/main_app_window.py` confirms `self._browser = QWebEngineView()` with no explicit profile — default global profile — localStorage persists across sessions.
9. **el.constructionCanvas CSS:** The container holding the canvas must have `position: relative` for the absolutely-positioned dot overlay to render correctly. Add to forge.css or inline style.

## Implementation Notes

- `AppServices._initialize()` uses `threading.Lock` with a double-check before and inside the lock. See `app/services.py` — the implementation is correct. Do not refactor without understanding the double-check pattern (first check avoids lock contention on every call post-init; second check inside lock handles the race where two threads both pass the first check).
- The boot shell placeholder (`canvas-boot-shell`) is rendered by `renderBootShell()` only on first launch (no localStorage state). On subsequent launches, `scheduleInitialHydration()` runs with pre-hydrated `facilityState` and overwrites `el.constructionCanvas` directly with the live canvas.
- `runtime_or_503()` in `app/api/routes.py` applies to all data routes (`/api/runs`, `/api/validate-scenario`, `/api/runs/{id}`, `/api/reports/*`). The `/api/health` and `/forge` routes intentionally do NOT go through `runtime_or_503()` — they must always respond regardless of service state.
